# 工业级多模态 RAG 知识库聚合助手：前端开发指南

本文面向前端开发人员，基于以下技术栈构建可扩展、可维护、可观测的 AI 交互界面：

- `Vue 3`（`<script setup lang="ts">`）
- `TypeScript`
- `Vite 6`
- `Pinia`
- `Tailwind CSS`
- `Shadcn-vue`
- `Axios` + `@microsoft/fetch-event-source`

术语对齐约定（与后端保持一致）：

- 文档解析状态：`PENDING`、`PARSING`、`PARSED`、`FAILED`。
- SSE 事件类型：`start`、`delta`、`citation`、`done`、`error`、`ping`。
- 会话与消息领域模型：`sessions`、`messages`，避免混用别名。

目录约定建议：

- `components/ui` 仅放原子组件，不放业务逻辑。
- `composables` 处理可复用交互流程，不直接耦合页面。
- `stores` 专注状态，不承担网络请求细节。
- `api/modules` 专注数据访问，返回强类型对象。

## 3. 状态管理（Pinia）

推荐开启持久化插件（只持久化必要字段）。

### 3.1 `userStore`

职责：用户信息、Token、登录状态。

```ts
// src/stores/user.ts
import { computed, ref } from 'vue'
import { defineStore } from 'pinia'

export const useUserStore = defineStore('user', () => {
	const token = ref<string | null>(null)
	const profile = ref<{ id: string; name: string; email: string } | null>(null)
	const isLoggedIn = computed(() => Boolean(token.value))

	function setAuth(nextToken: string, nextProfile: NonNullable<typeof profile.value>) {
		token.value = nextToken
		profile.value = nextProfile
	}

	function clearAuth() {
		token.value = null
		profile.value = null
	}

	return { token, profile, isLoggedIn, setAuth, clearAuth }
})
```

### 3.4 `chatStore`

职责：消息数组、流式接收状态、加载状态。

```ts
// src/stores/chat.ts
import { ref } from 'vue'
import { defineStore } from 'pinia'

export interface ChatMessage {
	id: string
	role: 'user' | 'assistant' | 'system'
	content: string
	sources?: Array<{ title: string; chunkId: string }>
}

export const useChatStore = defineStore('chat', () => {
	const messages = ref<ChatMessage[]>([])
	const streaming = ref(false)
	const loading = ref(false)

	function appendMessage(msg: ChatMessage) {
		messages.value.push(msg)
	}

	function patchAssistantContent(id: string, chunk: string) {
		const target = messages.value.find((m) => m.id === id)
		if (target) target.content += chunk
	}

	return { messages, streaming, loading, appendMessage, patchAssistantContent }
})
```

## 4. API 请求封装

## 4.1 Axios 请求层

目标：统一 Token 注入、错误提示、全局 Loading。

```ts
// src/api/client.ts
import axios from 'axios'
import { useUserStore } from '@/stores/user'

export const http = axios.create({
	baseURL: '/api',
	timeout: 30000,
})

http.interceptors.request.use((config) => {
	const userStore = useUserStore()
	if (userStore.token) {
		config.headers.Authorization = `Bearer ${userStore.token}`
	}
	return config
})

http.interceptors.response.use(
	(resp) => resp.data,
	(error) => {
		const message = error?.response?.data?.message || '请求失败，请稍后重试'
		// 这里可接入 toast 系统
		console.error(message)
		return Promise.reject(error)
	},
)
```

全局 Loading 建议：

- 使用 `requestId` 计数方式，避免并发请求时闪烁。
- 在拦截器内统一 `+1/-1`，在根布局展示全局加载条。

### 4.2 SSE 工具封装

推荐 `@microsoft/fetch-event-source`，可控性高，支持 Header/POST。

```ts
// src/api/sse.ts
import { fetchEventSource } from '@microsoft/fetch-event-source'

interface StreamOptions {
	url: string
	token?: string
	payload: unknown
	onDelta: (text: string) => void
	onSource?: (source: unknown) => void
	onDone?: () => void
	onError?: (err: unknown) => void
}

export async function startChatStream(options: StreamOptions) {
	let retryCount = 0

	await fetchEventSource(options.url, {
		method: 'POST',
		headers: {
			'Content-Type': 'application/json',
			...(options.token ? { Authorization: `Bearer ${options.token}` } : {}),
		},
		body: JSON.stringify(options.payload),
		onopen(resp) {
			if (!resp.ok) throw new Error(`SSE open failed: ${resp.status}`)
		},
		onmessage(ev) {
			if (ev.event === 'done') {
				options.onDone?.()
				return
			}
			if (ev.event === 'source') {
				options.onSource?.(safeParse(ev.data))
				return
			}
			const data = safeParse(ev.data)
			if (typeof data?.delta === 'string') {
				options.onDelta(data.delta)
			}
		},
		onerror(err) {
			retryCount += 1
			if (retryCount > 5) {
				options.onError?.(err)
				throw err
			}
		},
	})
}

function safeParse(raw: string) {
	try {
		return JSON.parse(raw)
	} catch {
		return raw
	}
}
```

SSE 实践要点：

- 事件类型分流：`start`、`delta`、`citation`、`done`、`error`、`ping`。
- 指数退避重连：`1s/2s/4s/8s`。
- 页面切换时主动取消流，防止内存泄漏。

### 5.2 主聊天界面

#### 5.2.1 消息列表渲染（Markdown + Shiki）

```ts
// src/utils/markdown.ts
import MarkdownIt from 'markdown-it'
import { createHighlighter } from 'shiki'

let highlighterPromise: ReturnType<typeof createHighlighter> | null = null

export async function renderMarkdown(content: string) {
	if (!highlighterPromise) {
		highlighterPromise = createHighlighter({
			themes: ['github-light', 'github-dark'],
			langs: ['ts', 'js', 'json', 'bash', 'python', 'java'],
		})
	}
	const highlighter = await highlighterPromise
	const md = new MarkdownIt({
		html: false,
		highlight(code, lang) {
			const language = lang || 'text'
			return highlighter.codeToHtml(code, { lang: language, themes: { light: 'github-light', dark: 'github-dark' } })
		},
	})
	return md.render(content)
}
```

复制按钮建议：

- 使用事件委托监听代码块按钮。
- 复制成功后显示 `Copied` 状态 1.5 秒。

#### 5.2.2 流式消息展示与自动滚动

基于 `@vueuse/core` 的 `useScroll`：

```ts
import { useScroll } from '@vueuse/core'

const containerRef = ref<HTMLElement | null>(null)
const { y } = useScroll(containerRef)

function scrollToBottom() {
	if (!containerRef.value) return
	y.value = containerRef.value.scrollHeight
}
```

流式更新时机：

- 收到 `delta` 立即追加内容。
- `nextTick + requestAnimationFrame` 后滚动到底部。
- 用户手动上滑时暂停自动滚动，避免抢焦点。

## 5.3 知识库管理页面

页面功能：

- 知识库切换（下拉/侧边导航）。
- 文档列表（名称、状态、进度、更新时间）。
- 上传组件（拖拽上传、多文件队列）。
- 删除操作（二次确认 + 乐观更新 + 回滚）。

推荐状态字段（前端展示）与后端枚举映射：

- `uploading`（仅前端本地态） -> 尚未提交后端
- `PENDING` -> 已上传，待解析
- `PARSING` -> 解析或向量化处理中
- `PARSED` -> 可检索
- `FAILED` -> 处理失败

### 5.4 历史会话侧边栏

能力要求：

- 会话列表分页或按需加载。
- 新建会话（默认标题可由首条消息生成）。
- 删除会话（若删除当前会话，自动切换到下一条）。
- 切换会话时取消当前 SSE，防止串流写入错误会话。

## 6. 关键交互逻辑

### 6.1 流式响应特殊 token 处理

建议后端通过结构化事件传输：

- `event: delta`，`data: {"delta":"..."}`
- `event: citation`，`data: {"title":"文档A","chunkId":"c_1"}`
- `event: done`

如果后端返回文本 token（如 `[SOURCE:id]`），前端策略：

1. 先在 `utils/tokenParser.ts` 解析并剥离特殊 token。
2. 正文追加到气泡内容。
3. 引用信息写入 `message.sources`，用于卡片渲染。

### 6.2 文件上传到向量化全流程

推荐流程：

1. 调用上传接口，获取 `fileId`。
2. 调用解析任务创建接口，获取 `taskId`。
3. 前端轮询 `taskId` 状态：`PENDING -> PARSING -> PARSED`。
4. 解析成功后触发向量化（若后端未自动触发）。
5. 实时更新文档进度条，并在成功后刷新文档列表。

轮询建议：

- 间隔 `2s`，最大 `5min` 超时。
- 页面卸载或用户离开时取消轮询。

### 6.3 语音输入流程

实现方式：

- 浏览器原生 `MediaRecorder` 录音。
- 录音 Blob 上传至 Whisper 接口。
- 返回文本自动填充输入框。

关键点：

- 首次录音前检查麦克风权限。
- 限制单次录音时长（如 60 秒）。
- 音频上传过程显示状态（录音中/转写中/完成）。

## 8. 本地存储与安全性

### 8.1 Token 存储策略

推荐优先级：

1. `httpOnly + secure + sameSite` Cookie（更安全，防 XSS 窃取）。
2. 内存 Token + 短期刷新机制（刷新后失效，安全性较高）。
3. `localStorage`（实现简单，但有 XSS 风险）。

如果必须使用 `localStorage`：

- 严格 CSP。
- 输入输出统一转义。
- 对富文本渲染做白名单过滤。

### 8.2 敏感文件访问控制

建议：

- 所有文件访问必须带鉴权并做服务端权限校验。
- 前端仅展示当前用户有权限的数据。
- 下载链接使用短时效签名 URL。
- 上传后文件名与路径脱敏，不在前端暴露真实存储路径。

## 9. 交付检查清单

上线前至少完成以下验证：

- 登录态失效后自动跳转登录。
- SSE 中断可重连，且不会重复拼接消息。
- 多文件上传进度准确，失败可重试。
- 大量历史消息下滚动与输入不卡顿。
- 深色/浅色主题切换无闪烁。
- XSS 基础防护生效（Markdown 渲染安全策略通过）。

以上规范可作为项目一期的前端落地基线。建议在迭代中补充 E2E 自动化测试（Playwright）和核心链路埋点（会话成功率、平均响应时延、上传成功率），确保 AI 应用在复杂生产环境中的稳定性。
