## 4. Bug #1：Milvus 初始化逻辑缺失实际调用

### 4.1 现象

`VectorStoreServiceImpl` 在 Collection 不存在时，创建了参数对象，但没有执行 Milvus 客户端 API。  
结果是系统“看起来执行了初始化”，但 Collection/Index 实际不存在，后续 upsert/retrieve 失败或异常。

### 4.2 根因

属于典型“参数构建完成但漏掉执行调用”的实现缺陷，且日志不足以暴露“未真正提交到 Milvus”。

### 4.3 修复方案（最终版）

在初始化流程中补齐真实 API 调用，并统一使用 `handleResult(...)` 做结果校验，失败即抛错阻断启动流程。

### 4.4 关键代码（可直接复制）

```java
CreateCollectionParam createParam = CreateCollectionParam.newBuilder()
        .withCollectionName(collectionName)
        .withSchema(schema)
        .build();

// 【修复】实际执行创建 Collection
R<RpcStatus> createResult = milvusClient.createCollection(createParam);
handleResult(createResult, "创建 Collection");
log.info("[Milvus] Collection '{}' 创建成功", collectionName);

// 创建索引
createIndex();
```

```java
CreateIndexParam indexParam = CreateIndexParam.newBuilder()
        .withCollectionName(collectionName)
        .withFieldName("embedding")
        .withIndexType(IndexType.IVF_FLAT)
        .withMetricType(MetricType.valueOf(metricType))
        .withExtraParam("{\"nlist\": 1024}")
        .build();

// 【修复】实际执行创建索引
R<RpcStatus> result = milvusClient.createIndex(indexParam);
handleResult(result, "创建向量索引");
log.info("[Milvus] 向量索引创建成功");
```

### 4.5 为什么这样做

- 最小改动解决根因，不引入额外行为分叉。
- 保持现有 `handleResult` 错误语义，符合当前代码风格。
- 失败时快速暴露问题，避免“服务启动成功但检索不可用”的隐蔽故障。

### 4.6 重难点与解决

**难点：** Milvus 初始化步骤多，任何一步缺失都会造成链路不可用。  
**解决：** 将“创建 Collection → 创建 Index → Load Collection”串成严格顺序，并在每步校验返回码。

## 5. Bug #2：文档处理事务失效（Spring AOP 同类调用）

### 5.1 现象

原流程在 `@Async` 方法内部调用同类 `@Transactional` 方法，事务代理未生效。  
一旦处理中途失败，PostgreSQL 写入可能已部分落库，出现状态与数据不一致。

### 5.2 根因

Spring AOP 基于代理，同类内部方法调用不会走代理链，`@Transactional` 失效。

### 5.3 修复方案（最终版）

采用**职责拆分**（生产推荐做法）：

1. `DocumentProcessServiceImpl`：仅负责异步调度、状态校验、重试控制。
2. 新建 `DocumentProcessTransactionService` + `...Impl`：承载完整事务处理流程。
3. 异步服务通过 Spring 注入调用事务服务，确保进入代理链，事务生效。

### 5.4 新增文件

- `rag-knowledge/src/main/java/com/guiderag/knowledge/service/DocumentProcessTransactionService.java`
- `rag-knowledge/src/main/java/com/guiderag/knowledge/service/impl/DocumentProcessTransactionServiceImpl.java`

### 5.5 关键代码（可直接复制）

**异步调度层：**

```java
@Async("documentProcessExecutor")
@Override
public void processDocumentAsync(Long documentId) {
    Document doc = documentMapper.selectByIdOnly(documentId);
    if (doc == null) {
        log.error("[DocumentProcess] ❌ 文档不存在: documentId={}", documentId);
        return;
    }

    String status = doc.getParseStatus();
    if (!"PENDING".equals(status) && !"FAILED".equals(status)) {
        log.warn("[DocumentProcess] ⚠ 跳过处理，状态不符: documentId={}, status={}", documentId, status);
        return;
    }

    if ("FAILED".equals(status) && doc.getRetryCount() >= MAX_RETRY_COUNT) {
        log.warn("[DocumentProcess] ⚠ 已达最大重试次数，跳过: documentId={}, retryCount={}",
                documentId, doc.getRetryCount());
        return;
    }

    // 【关键】跨 Service 调用，触发事务代理
    transactionService.processDocumentInTransaction(doc);
}
```

**事务处理层：**

```java
@Override
@Transactional(rollbackFor = Exception.class)
public void processDocumentInTransaction(Document doc) {
    Long documentId = doc.getId();
    try {
        documentMapper.updateParseStatus(documentId, "PARSING", null);

        String textContent;
        try (InputStream inputStream = minioClient.getObject(
                GetObjectArgs.builder()
                        .bucket(doc.getMinioBucket())
                        .object(doc.getMinioObjectKey())
                        .build()
        )) {
            textContent = documentParserService.parseDocument(inputStream, doc.getFileName());
        }

        List<DocumentChunk> chunks = chunkingService.chunkDocument(doc.getTenantId(), documentId, textContent);
        List<String> texts = chunks.stream().map(DocumentChunk::getContentText).toList();
        List<float[]> embeddings = embeddingService.embedBatch(texts);

        if (embeddings.size() != chunks.size()) {
            throw new BizException("B0001",
                    String.format("向量数量与分块数量不匹配: %d vs %d", embeddings.size(), chunks.size()));
        }

        for (DocumentChunk chunk : chunks) {
            chunk.setId(IdUtil.getSnowflakeNextId());
            documentChunkMapper.insert(chunk);
        }

        List<VectorStoreService.VectorEntry> vectorEntries = new ArrayList<>();
        for (int i = 0; i < chunks.size(); i++) {
            DocumentChunk chunk = chunks.get(i);
            vectorEntries.add(VectorStoreService.VectorEntry.builder()
                    .vectorId(chunk.getVectorId())
                    .tenantId(chunk.getTenantId())
                    .documentId(chunk.getDocumentId())
                    .chunkIndex(chunk.getChunkIndex())
                    .contentText(chunk.getContentText())
                    .embedding(embeddings.get(i))
                    .build());
        }
        vectorStoreService.upsertBatch(vectorEntries);

        documentMapper.updateParseStatusSuccess(documentId, "PARSED", chunks.size(), LocalDateTime.now());
    } catch (Exception e) {
        documentMapper.updateParseStatusFailed(documentId, "FAILED", e.getMessage(), doc.getRetryCount() + 1);
        throw new RuntimeException("文档处理失败: " + e.getMessage(), e);
    }
}
```

### 5.6 为什么这样做

- 符合微服务与分层最佳实践：调度与事务分离、边界清晰。
- 对 Spring AOP 机制友好，不依赖不推荐的“自注入/暴露代理”等技巧。
- 后续扩展（重试、补偿、监控）更容易落地。

### 5.7 重难点与解决

1. **难点：Milvus 无本地事务**  
   解决：以 PostgreSQL 事务为主，Milvus 通过状态机与重试实现最终一致性（`PENDING -> PARSING -> PARSED/FAILED`）。

2. **难点：失败路径可观测性**  
   解决：失败时更新 `FAILED + error_message + retry_count`，便于定位和运营重试。

3. **难点：重试幂等性**  
   解决：限制最大重试次数，且向量写入走批量 upsert 路径，避免无界重复插入。

## 6. Bug #3：多轮对话缺失历史上下文

### 6.1 现象

同步与流式对话均只传 `SystemMessage + 当前UserMessage`，未注入历史消息。  
用户第二问开始，模型无法理解“它/这个/刚才那个”等上下文指代。

### 6.2 根因

消息存储存在，但 ChatService 在调用 LangChain4j 时没有读取并拼装历史上下文。

### 6.3 修复方案（最终版）

1. 在 `MessageMapper` / `MessageMapper.xml` 新增“按会话查询最近历史消息”接口。
2. 在 `ChatServiceImpl` 新增 `getHistoryMessages(sessionId, tenantId)`。
3. 对同步 `chat(...)` 与流式 `chatStream(...)` 同步接入历史上下文。
4. 加入“双阈值控制”：最大历史条数 + 最大 token 预算，防止超窗。

### 6.4 关键代码（可直接复制）

**Mapper 接口：**

```java

        @Param("tenantId") Long tenantId,
```

```xml
    FROM message
    WHERE session_id = #{sessionId}
```

**ChatService 历史消息组装：**

```java
private static final int MAX_HISTORY_MESSAGES = 10;
private static final int MAX_HISTORY_TOKENS = 2000;

private List<dev.langchain4j.data.message.ChatMessage> getHistoryMessages(Long sessionId, Long tenantId) {
    List<Message> historyList = messageMapper.selectRecentBySessionId(sessionId, tenantId, MAX_HISTORY_MESSAGES);
    if (historyList == null || historyList.isEmpty()) {
        return Collections.emptyList();
    }

    // DB倒序 -> LLM正序
    Collections.reverse(historyList);

    List<dev.langchain4j.data.message.ChatMessage> messages = new ArrayList<>();
    int totalTokens = 0;
    for (Message msg : historyList) {
        int msgTokens = estimateTokens(msg.getContent());
        if (totalTokens + msgTokens > MAX_HISTORY_TOKENS) {
            break;
        }
        totalTokens += msgTokens;

        switch (msg.getRole()) {
            case "user" -> messages.add(UserMessage.from(msg.getContent()));
            case "assistant" -> messages.add(AiMessage.from(msg.getContent()));
            default -> { /* 仅保留会话主干角色 */ }
        }
    }
    return messages;
}
```

**同步对话注入历史：**

```java

List<dev.langchain4j.data.message.ChatMessage> allMessages = new ArrayList<>();
```

**流式对话注入历史：**

```java
List<dev.langchain4j.data.message.ChatMessage> historyMessages = getHistoryMessages(sessionId, userId);
List<dev.langchain4j.data.message.ChatMessage> allMessages = new ArrayList<>();
allMessages.add(SystemMessage.from(systemPrompt));
allMessages.addAll(historyMessages);
allMessages.add(UserMessage.from(dto.getQuestion()));

streamingChatModel.generate(allMessages, handler);
```

### 6.5 为什么这样做

- 让会话真正具备“连续记忆”，恢复对话产品的基础能力。
- 同步与流式统一策略，避免两个入口行为不一致。
- 限流与预算并用，兼顾效果与稳定性。

### 6.6 重难点与解决

1. **难点：上下文窗口受模型限制**  
   解决：条数上限（10）+ 预算上限（2000 tokens）双重控制。

2. **难点：消息顺序与模型理解**  
   解决：SQL 倒序取“最近消息”，业务层反转后按时间正序喂给 LLM。

3. **难点：角色污染**  
   解决：SQL 仅保留 `user/assistant`，规避 tool/system 杂讯干扰主对话链。

## 7. 与 Knife4j / API 契约的一致性说明

本次修复未新增/删除外部 API 路径，也未破坏现有请求/响应结构，属于**服务内部逻辑增强**：

- `rag-knowledge`：文档处理链路稳定性增强（事务生效 + 状态一致性）
- `rag-chat`：回答生成上下文增强（多轮能力）

因此 Knife4j 对外文档结构无需破坏性调整。建议在接口说明中补充两条“行为语义”：

1. 对话接口为“多轮上下文感知”。
2. 文档解析失败会进入可重试状态并记录失败原因。
