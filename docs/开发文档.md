# 工业级多模态 RAG 知识库聚合助手｜项目开发文档

## 1. 项目概述

### 1.1 背景
企业内部知识分散在 PDF、Word、图片、录音、工单系统和 Wiki 中，传统关键词检索存在以下问题：
1. 语义检索能力弱，难以理解同义表达。
2. 多模态内容（图片、语音）无法统一检索与问答。
3. 知识更新后，人工维护 FAQ 成本高。
4. 大模型直接回答容易出现幻觉，缺少企业知识约束。

### 1.2 业务痛点
1. 知识孤岛：文档格式多样，结构不统一。
2. 检索召回不准：仅全文检索导致噪声多。
3. 响应时延高：复杂推理链路下用户等待时间长。
4. 合规可追溯不足：回答来源和引用证据不透明。
5. 高并发下资源浪费：线程阻塞导致吞吐受限。

### 1.3 解决方案：多模态 RAG
采用多模态解析 + 向量检索 + 重排序 + LLM 生成 + 流式返回的端到端架构：
1. 文档、图片、语音统一入库。
2. 多模型生成 Embedding（文本、图像、语音转文本后向量化）。
3. 向量检索召回 + Reranker 精排。
4. 拼接可追溯上下文 Prompt，约束大模型回答。
5. 通过 SSE 实时推送生成内容与引用来源。

### 1.4 核心功能
1. 多格式知识导入：PDF、DOCX、PPT、图片、音频。
2. 智能问答：基于企业知识库的上下文增强问答。
3. 语音交互：录音上传，Whisper 转写后进入同一问答链路。
4. 图像理解：图像内容解析后纳入检索上下文。
5. 证据可追溯：回答附带来源文档、页码、切片 ID。
6. 实时流式体验：SSE 增量输出，前端逐字渲染。
7. 多租户和多知识库隔离：权限与数据隔离可配置。

### 1.5 文档阅读主线
为避免实现阶段出现“架构设计与接口约定脱节”，本文按以下主线展开：
1. 先明确技术选型与约束边界（第 2 章）。
2. 再定义系统分层与核心链路（第 3 章）。
3. 随后落到存储模型与接口规范（第 4、5 章）。
4. 最后给出业务流程、部署与微服务演进建议（第 6、7、8 章）。

该顺序可保证从“为什么这样做”过渡到“具体怎么做”，减少跨团队理解成本。

## 2. 技术选型分析

### 2.1 前端技术栈
1. Vue 3 + Script Setup + TypeScript
	- 理由：组合式 API 适合复杂状态拆分，TypeScript 强类型降低维护成本。
	- 建议版本：`vue@3.5+`，`typescript@5.6+`。

2. Vite 6
	- 理由：冷启动快、HMR 高效，适合中大型前端工程。
	- 建议版本：`vite@6.x`。

3. Pinia
	- 理由：官方状态管理，类型推导好，适合会话状态与流式消息状态管理。

4. Tailwind CSS + Shadcn-vue + Lucide Vue Next
	- 理由：高效率构建一致 UI 体系，组件可定制，图标轻量。

5. Markdown-it + Shiki
	- 理由：支持模型回答 Markdown 渲染和代码高亮，提升可读性。

6. VueUse
	- 理由：常用组合函数（节流、防抖、网络状态）减少重复代码。

7. @microsoft/fetch-event-source（SSE）
	- 理由：浏览器原生事件流语义，适合服务端单向推送 token 流。

### 2.2 后端技术栈
1. Spring Boot 3.4+ + Java 21
	- 理由：现代化生态、可观测性成熟、与企业体系兼容强。

2. Java 虚拟线程（Project Loom）
	- 优势：对 I/O 密集任务（检索、对象存储、模型 API 调用）可显著提升并发处理能力。
	- 价值：
	  1. 传统平台线程在阻塞 I/O 时占用系统线程，成本高。
	  2. 虚拟线程使每请求一线程模型可行，编码模型仍保持同步直观。
	  3. 降低异步回调复杂度，提升可维护性。
	- 典型场景：文档解析任务、并行召回、多模型路由、SSE 长连接请求。

3. LangChain4j
	- 理由：快速接入 LLM、Embedding、工具调用和 RAG 链路编排。
	- 选型建议：
	  - 偏 Spring 生态与可观测性：Spring AI。
	  - 偏灵活链路编排：LangChain4j。

4. Apache Tika / PDFBox + Docling Java Wrapper
	- 理由：覆盖主流文档解析，复杂版面抽取能力强。

5. Spring Security + JWT
	- 理由：成熟鉴权框架，支持无状态 API 和角色权限控制。

### 2.3 AI 与模型
1. Ollama（本地 Llama 3 / DeepSeek-V3）
	- 理由：本地化部署降低敏感数据外发风险，支持离线场景。

2. DeepSeek API / OpenAI GPT-4o（云端）
	- 理由：复杂推理、图像理解能力强，作为高质量 fallback。

3. BGE-M3（Embedding）
	- 理由：多语种和检索表现优秀，适配企业中文场景。

4. BGE-Reranker-v2
	- 理由：对初召回结果重排，提升答案相关性与准确率。

5. Whisper
	- 理由：语音转文字稳定，支持多语言。

6. CLIP / GPT-4o-mini（图像理解）
	- 理由：可提取图像语义用于检索或直接视觉问答增强。

### 2.4 数据与基础设施
1. Milvus / Qdrant
	- 理由：专业向量检索能力、索引丰富、可扩展性好。

2. PostgreSQL
	- 理由：强一致关系数据管理，事务可靠，生态成熟。

3. Redis
	- 理由：低延迟缓存，会话上下文和限流计数器高效。

4. MinIO
	- 理由：S3 兼容对象存储，适合原始文件与解析产物管理。

5. Prometheus + Grafana
	- 理由：可观测体系标准组合，便于延迟、吞吐、Token 成本监控。

### 2.5 为什么 SSE 优于 WebSocket（本场景）
1. 问答流式输出是服务端单向推送，SSE 天然匹配。
2. 基于 HTTP，穿透网关和负载均衡更简单，调试成本更低。
3. 自动重连语义清晰（`Last-Event-ID`），实现成本低。
4. 相比 WebSocket，协议和心跳维护更轻量。
5. 若未来需要双向实时协作，再补充 WebSocket 通道即可。

## 3. 系统架构图（文字分层）

### 3.1 分层架构
1. 接入层（Frontend/API Gateway）
	- Vue3 前端、Nginx 或 API Gateway、JWT 鉴权入口、限流与审计。

2. 业务逻辑层（Spring Boot）
	- 用户与权限、知识库管理、文档管理、会话管理、任务调度、SSE 输出控制。

3. AI 编排层（RAG Orchestrator）
	- 文档解析、切片策略、Embedding 生成、向量检索、Rerank、Prompt 构建、模型路由、本地和云模型切换。

4. 数据层
	- PostgreSQL（元数据）
	- Milvus/Qdrant（向量）
	- Redis（会话缓存和限流）
	- MinIO（原始文件与中间产物）

5. 基础设施层
	- Docker Compose、Prometheus、Grafana、日志系统（ELK 或 OTel 可选）。

### 3.2 关键交互流程（示例）
用户上传 PDF -> API 接收并鉴权 -> MinIO 存原始文件 -> 异步解析（Tika/PDFBox/Docling）-> 文本切片 -> Embedding（BGE-M3）-> 向量入 Milvus/Qdrant + 元数据入 PostgreSQL -> 用户提问 -> Query Embedding -> 向量召回 TopK -> Reranker 精排 -> 构造 Prompt（含来源片段）-> LLM 流式生成 -> SSE 推送前端 -> 前端 Markdown 实时渲染 + 引用来源展示。

### 4.2 向量数据库设计（Milvus/Qdrant）

#### 4.2.1 Collection 建议（统一多模态）
- Collection：`kb_chunks`（按知识库逻辑隔离，物理可单集合 + 过滤）
- 主键：`chunk_uid`（字符串或 int64）
- 向量字段：
  1. `text_vector`：`FLOAT_VECTOR(1024)`（BGE-M3）
  2. `image_vector`：`FLOAT_VECTOR(768)`（CLIP）
  3. `audio_text_vector`：`FLOAT_VECTOR(1024)`（Whisper 转文本后 Embedding）
- 标量字段：
  1. `kb_id`（INT64）
  2. `doc_id`（INT64）
  3. `modality`（VARCHAR）
  4. `chunk_index`（INT32）
  5. `page_no`（INT32）
  6. `language`（VARCHAR）
  7. `created_at`（INT64 时间戳）

#### 4.2.2 索引建议
1. Milvus：HNSW 或 IVF_FLAT（中高精度场景优先 HNSW）。
2. Qdrant：HNSW 默认 + payload filter。
3. 度量：文本常用 COSINE，按模型说明配置。
4. 检索策略：先 filter（`kb_id`、`modality`）后 ANN 检索。

#### 4.2.3 多模态 Embedding 存储策略
1. 同一 `chunk_uid` 可写入多个向量字段（按模态补齐）。
2. 查询时根据用户输入模态选择对应向量字段检索。
3. 融合策略：
	- 早期融合：多向量加权召回。
	- 后期融合：分别召回后合并去重再 Rerank（推荐，易调优）。

### 4.3 Redis 数据结构设计
1. 会话上下文缓存
	- Key：`chat:ctx:{sessionId}`
	- Type：List 或 Stream
	- TTL：30 到 120 分钟
	- 内容：最近 N 轮消息摘要与裁剪上下文

2. 限流计数器
	- Key：`rate:user:{userId}:{yyyyMMddHHmm}`
	- Type：String（INCR）
	- 结合 Lua 保证原子限流

3. 异步任务状态
	- Key：`task:parse:{docId}`
	- Type：Hash
	- 字段：`status`、`progress`、`error`、`updatedAt`

4. SSE 会话临时状态
	- Key：`sse:session:{sessionId}`
	- Type：Hash
	- 字段：`lastEventId`、`model`、`startedAt`

### 4.4 MinIO 存储桶结构
1. Bucket：`rag-raw`（原始文件）
	- 路径：`/{tenantId}/{kbId}/{docId}/original/{filename}`

2. Bucket：`rag-processed`（解析中间产物）
	- 路径：`/{tenantId}/{kbId}/{docId}/parsed/{page_or_chunk}.json`

3. Bucket：`rag-audio`（音频临时）
	- 路径：`/{tenantId}/{sessionId}/{messageId}/audio.wav`

4. Bucket：`rag-image`（图片抽取与缩略图）
	- 路径：`/{tenantId}/{kbId}/{docId}/images/{index}.png`

建议：开启版本控制与生命周期策略（如 30 天清理临时音频）。

## 5. 接口设计规范

### 5.1 RESTful 风格
1. 资源名复数：`/rag/v1/documents`
2. HTTP 语义：
	- `GET` 查询
	- `POST` 创建或触发任务
	- `PUT/PATCH` 更新
	- `DELETE` 删除
3. 幂等要求：上传和任务触发需防重（`checksum` 或幂等键）。

### 5.2 统一响应结构
```json
{
  "code": 0,
	"message": "OK",
  "data": {},
  "traceId": "8f3d9d..."
}
```
- `code=0` 表示成功，非 0 为业务错误码。
- 建议错误码分段：1000 参数，2000 鉴权，3000 知识库，4000 模型。

### 5.3 分页规范
请求参数：
- `page`（从 1 开始）
- `size`（默认 20，最大 100）
- `sort`（如 `createdAt,desc`）

响应示例：
```json
{
  "code": 0,
	"message": "OK",
  "data": {
	 "items": [],
	 "page": 1,
	 "size": 20,
	 "total": 135
  }
}
```

### 5.4 全局异常处理
1. 使用 `@RestControllerAdvice` 统一拦截。
2. 错误响应统一结构，不向前端暴露堆栈细节。
3. 记录 `traceId` 到日志便于链路排查。

### 5.5 JWT 鉴权机制
1. 登录后签发 `accessToken`（短期）与 `refreshToken`（长期）。
2. Header：`Authorization: Bearer <token>`
3. 权限控制：基于角色 + 资源归属（`kb_id`）二次校验。
4. 黑名单和注销：可用 Redis 存储失效令牌标记。

### 5.6 SSE 流式接口规范
接口示例：`POST /rag/v1/chat/stream`

响应头：
- `Content-Type: text/event-stream`
- `Cache-Control: no-cache`
- `Connection: keep-alive`

消息格式（`data:` + 双换行分隔）：
```text
event: delta
id: 101
data: {"delta":"你好","finishReason":null}

event: citation
data: {"docId":12,"chunkUid":"ck_001","score":0.92}

event: done
data: {"messageId":998,"finishReason":"stop"}

```

建议：
1. 心跳包：每 15 到 30 秒发送 `event: ping`。
2. 断线续传：支持 `Last-Event-ID`。
3. 结束标记：统一 `event: done`。
4. 事件约定：推荐统一使用 `start/delta/citation/done/error/ping`，避免不同模块使用 `token/chunk/message` 造成前端分支膨胀。

## 6. 核心业务流程

### 6.1 多格式文档上传与解析流程
1. 前端上传文件并携带 `kbId`。
2. 后端鉴权后写入 MinIO，创建 `document` 记录（`PENDING`）。
3. 异步任务消费：
	- PDF/Office：Tika/PDFBox/Docling 提取文本与结构
	- 图片：OCR 或视觉模型提取描述文本
	- 音频：Whisper 转写文本
4. 文本清洗与切片（如 500 到 1000 tokens，重叠 50 到 100）。
5. 生成 Embedding（BGE-M3 或 CLIP）。
6. 向量写入 Milvus/Qdrant，元数据写入 PostgreSQL。
7. 更新文档状态为 `PARSED`，失败则记录错误并可重试。

### 6.2 用户提问流式应答流程
1. 接收问题，校验会话与知识库权限。
2. Query 向量化并在向量库按 `kbId` 过滤召回 TopK（如 30）。
3. Reranker 精排到 TopN（如 5 到 8）。
4. 构造 Prompt：
	- 系统指令（禁止编造，必须基于上下文）
	- 用户问题
	- 检索证据片段（附来源）
5. 调用 LLM 进行流式生成。
6. 后端将 token 增量通过 SSE 推送。
7. 结束后落库 `message`（内容、token、耗时、引用）。

补充说明：该链路默认由 RAG 编排服务统一调度，检索、重排、生成可作为内部模块或独立微服务。若进入 Spring Cloud 微服务架构，建议保持 API 协议不变，仅调整服务边界与调用方式。

### 6.3 语音交互流程
1. 前端录音上传到 `/rag/v1/audio/transcriptions`。
2. 后端调用 Whisper 得到文本。
3. 文本进入与普通提问相同的 RAG 管道。
4. SSE 返回回答；可选 TTS 再转语音回放（后续扩展）。

## 8. Spring Cloud 微服务最佳实践符合度评估

### 8.1 当前方案符合项
1. 技术底座与微服务兼容：`Spring Boot 3.4+`、`Java 21`、`SSE`、可观测性组件均可平滑迁移至 Spring Cloud。
2. 业务边界已具备拆分基础：认证、知识库、文档处理、会话、RAG 编排逻辑相对独立。
3. 存储分层清晰：PostgreSQL、Milvus/Qdrant、Redis、MinIO 职责明确，符合微服务下的数据职责分离方向。

### 8.2 主要差距
1. 服务部署形态仍偏单体：当前 `app` 容器承载多类职责，不满足“独立扩缩容”最佳实践。
2. 缺少服务治理基础组件说明：未明确服务注册发现、配置中心、熔断限流、服务间鉴权与零信任边界。
3. 异步链路与一致性策略不完整：文档解析到向量入库缺少事件总线、重试与补偿设计说明。

### 8.3 推荐落地方案（Spring Cloud 2023.x / 2024.x）
1. 基础治理：强烈聚焦 **Nacos** 用于服务发现与动态配置管理，使用 `Spring Cloud Gateway` 作为统一入口。
2. 服务拆分建议：
	- `gateway-service`：统一入口、鉴权透传、Sentinel 全局限流与审计。
	- `auth-service`：用户与 JWT/OAuth2。
	- `kb-service`：知识库、文档元数据、权限。
	- `ingestion-service`：解析、切片、向量化任务编排。
	- `retrieval-service`：向量检索与重排。
	- `chat-orchestrator-service`：Prompt 组装、模型路由、SSE 输出。
3. 异步与削峰：引入 **RocketMQ** 作为事件总线，承载解析与向量化任务，并通过事务消息保证分布式数据最终一致。
4. 容错与稳定性：使用 **Sentinel**（流控、降级、系统保护）全面保护模型侧与核心路由节点的依赖稳定性。
5. 可观测性：采用 `Micrometer + OpenTelemetry + Prometheus + Grafana`，实现跨服务 Trace/Metric/Log 关联。

### 8.4 结论
当前文档描述的方案可作为“单体优先、逐步微服务化”的合理起点，但尚未达到 Spring Cloud 微服务最佳实现。按 8.3 完成治理组件与服务边界拆分后，可达到企业级可扩展、可观测、可治理的目标状态。
