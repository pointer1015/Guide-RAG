# 工业级多模态 RAG 知识库聚合助手 - 数据库设计文档

## 1. 设计目标与存储分层

本项目采用多存储协同架构：

- PostgreSQL：关系数据与事务一致性（租户、用户、知识库、文档、会话、消息、任务状态）。
- Milvus/Qdrant：向量检索（文本、图像，预留音频语义向量）。
- Redis：会话短期上下文缓存、限流、任务进度、分布式锁。
- MinIO：原始文件二进制存储（PDF/图片/音频）。

核心设计原则：

- 全链路多租户隔离：所有核心业务表包含 `tenant_id`。
- 元数据与对象分离：文件元数据在 PostgreSQL，文件内容在 MinIO。
- 检索可追溯：向量检索结果可回溯到 `document_chunks.id`。
- 可扩展多模态：文本与图像支持统一检索，并可按模态降级过滤。

与其他文档的衔接关系：

- 与《开发文档》对齐：本文定义的数据模型为 RAG 链路、SSE 回答落库与审计提供实体基础。
- 与《API 文档》对齐：API 中的 `sessions/messages/documents` 字段语义以本文表结构为准。
- 与《前端开发文档》对齐：前端状态展示字段应映射本文枚举与任务状态流转。

## 2. ER 图（文字描述）

### 2.1 主要实体

- `tenant`（租户）
- `user`（用户）
- `tenant_user`（租户-用户关系）
- `knowledge_base`（知识库）
- `document`（文档）
- `document_chunk`（文档块）
- `session`（聊天会话）
- `message`（消息）
- `retrieval_log`（检索日志，可审计）

### 2.2 实体关系

1. `tenant` 1:N `knowledge_base`  
	一个租户可拥有多个知识库。

2. `tenant` N:M `user`（通过 `tenant_user`）  
	用户可属于多个租户（企业组织场景）；个人用户也可对应一个默认租户。

3. `knowledge_base` 1:N `document`  
	文档归属某个知识库。

4. `document` 1:N `document_chunk`  
	文档解析后拆分为多个文本块/图像块并建立向量。

5. `user` 1:N `session`  
	用户可发起多个会话。

6. `session` 1:N `message`  
	会话包含完整消息历史（用户提问、AI回答、工具消息）。

7. `message` N:M `document_chunk`（逻辑层，物理可用 JSON 记录）  
	`message.referenced_chunk_ids` 存储一次回答引用的知识块 ID 列表。

8. `message` 1:N `retrieval_log`（可选）  
	每次检索召回明细可落日志，用于可解释与离线评估。

## 4. 向量数据库设计（Milvus/Qdrant）

## 4.1 Collection 规划建议

两种可选策略：

1. 全局统一 Collection（推荐默认）  
	优点：运维简单、索引维护成本低、跨知识库检索方便。  
	做法：在 `metadata` 中保存 `tenant_id`、`knowledge_base_id` 并在查询时强制过滤。

2. 按租户或知识库拆分 Collection  
	优点：物理隔离强、便于热点租户独立调优。  
	缺点：Collection 数量增多，管理与资源开销上升。

推荐：

- 中小规模（< 1000 租户）：统一 Collection + 强过滤。
- 大规模企业版：按租户分区或按租户拆 Collection。

## 4.2 统一向量结构

字段建议：

- `id`：向量主键（字符串或整型雪花ID）。
- `vector`：Embedding 向量。
- `metadata`：JSON，包含 `tenant_id`、`knowledge_base_id`、`document_id`、`chunk_id`、`chunk_index`、`modality`、`file_type`、`text`、`minio_object_key` 等。

多模态支持建议：

- 若使用同一多模态模型（如可将文本/图像映射到同空间），可统一一个向量字段做跨模态检索。
- 若文本与图像模型维度不同，建议同 Collection 多向量字段（Qdrant）或分 Collection（Milvus）并在应用层融合排序。

## 4.3 索引与度量

- 距离度量：`COSINE`（语义检索常用，兼容多数归一化向量）。
- Milvus：
  - 中等规模：`HNSW`（`M=16`, `efConstruction=200`）
  - 超大规模离线构建：`IVF_FLAT`（`nlist=1024~4096`）
- Qdrant：默认 HNSW，可调 `m`、`ef_construct`、`ef_search`。

## 4.4 Milvus (PyMilvus) 创建示例

```python
from pymilvus import connections, FieldSchema, CollectionSchema, DataType, Collection

connections.connect(alias="default", host="127.0.0.1", port="19530")

fields = [
	 FieldSchema(name="id", dtype=DataType.VARCHAR, is_primary=True, max_length=128),
	 FieldSchema(name="tenant_id", dtype=DataType.INT64),
	 FieldSchema(name="knowledge_base_id", dtype=DataType.INT64),
	 FieldSchema(name="document_id", dtype=DataType.INT64),
	 FieldSchema(name="chunk_id", dtype=DataType.INT64),
	 FieldSchema(name="modality", dtype=DataType.VARCHAR, max_length=32),
	 FieldSchema(name="vector", dtype=DataType.FLOAT_VECTOR, dim=1024),
]

schema = CollectionSchema(fields=fields, description="multimodal rag embeddings")
collection = Collection(name="rag_embeddings", schema=schema)

index_params = {
	 "metric_type": "COSINE",
	 "index_type": "HNSW",
	 "params": {"M": 16, "efConstruction": 200}
}
collection.create_index(field_name="vector", index_params=index_params)
collection.load()
```

注：复杂 `metadata` 推荐继续保存在 PostgreSQL 的 `document_chunk.metadata`，Milvus 中存必要过滤字段。

## 5. Redis 数据结构设计

## 5.1 会话上下文缓存

- Key：`session:{sessionId}:context`
- 类型：`String`（JSON 数组）或 `List`
- 建议：
  - 若经常整包读写，使用 `String` 存最近 N 条消息数组。
  - 若频繁 push/pop，使用 `List`，配合 `LPUSH + LTRIM` 保留最近 N 条。

示例（String）：

```text
SET session:987:context "[{\"role\":\"user\",\"content\":\"...\"}]" EX 3600
```

## 5.2 限流器（令牌桶）

- Key：`ratelimit:{tenantId}:{userId}:{api}`
- 实现：Lua 脚本原子扣减令牌，返回是否允许请求与剩余令牌。

Lua 逻辑要点：

- 根据 `last_refill_ts` 计算应补充 token。
- `tokens = min(capacity, tokens + refill)`。
- 当 `tokens >= cost` 时扣减并放行，否则拒绝。

## 5.3 临时状态

- 文档解析进度：`doc:{docId}:progress`（0-100）
- 文档阶段状态：`doc:{docId}:stage`（`uploaded/extracting/chunking/embedding/parsed/failed`）
- TTL：建议 24h，最终状态写回 PostgreSQL。

## 5.4 分布式锁

- Key：`lock:doc:{docId}:ingest`
- 命令：`SET lock:doc:123:ingest <uuid> NX PX 300000`
- 解锁：Lua 校验 value 后删除，避免误删他人锁。

## 6. MinIO 对象存储设计

## 6.1 Bucket 命名规范

可选策略：

1. 按用户隔离：`user-{userId}`  
	简单直观，但 bucket 数量可能过多。

2. 按租户隔离（推荐）：`tenant-{tenantId}`  
	更符合企业隔离边界，便于配额与生命周期管理。

推荐规则：

- 企业版：`tenant-{tenantId}`
- 个人版：可统一 `tenant-{defaultTenantId}`

## 6.2 对象路径规则

```text
/{knowledgeBaseId}/{documentId}/{filename}
```

例如：

```text
tenant-1001/2001/3001/product_manual_v2.pdf
tenant-1001/2001/3002/device_panel.png
tenant-1001/2001/3003/call_record_20260305.wav
```

## 6.3 元数据落位

- MinIO：只存二进制对象与少量对象标签。
- PostgreSQL：存权威元数据（大小、MIME、上传时间、hash、解析状态、归属关系）。

## 7. 索引与性能考虑

## 7.1 PostgreSQL 索引建议

高频过滤/排序字段：

- `document(tenant_id, knowledge_base_id, gmt_create DESC)`
- `document(parse_status, gmt_modified DESC)`
- `document_chunk(document_id, chunk_index)`
- `session(tenant_id, user_id, last_message_at DESC)`
- `message(session_id, gmt_create)`
- `message(tenant_id, gmt_create DESC)`
- JSONB 检索场景：`GIN(message.referenced_chunk_ids)`、`GIN(document_chunk.metadata)`

补充建议：

- 软删除场景可建部分索引：`WHERE is_deleted = 0`。
- 超大消息表建议按月分区（`gmt_create` range partition）。

## 7.2 向量索引调优

- HNSW：
  - `ef_search` 越大召回越高但延迟上升，在线建议 `64~256`。
  - `M` 越大内存越高，通常 `12~32`。
- IVF：
  - `nlist` 与数据量相关（可按 `sqrt(N)` 起步）。
  - 查询 `nprobe` 提升召回但增加计算量。
- 应做离线标注集评测，按 Recall@K 与 p95 latency 联合调参。

## 7.3 冷热数据分离

- 热数据：最近 7~30 天会话消息与活跃知识库向量，保留在高性能实例。
- 冷数据：归档消息、低频文档，迁移到低成本存储。
- 策略：
  - PostgreSQL：分区表 + 归档库。
  - 向量库：冷分区降副本或迁移到磁盘型节点。
  - Redis：仅缓存活跃会话，上线 TTL 自动淘汰。

## 8. 端到端数据流（上传到问答）

1. 用户上传 PDF/图片/音频 -> 文件写入 MinIO。  
2. `document` 写入元数据，`parse_status='PENDING'`。  
3. 异步任务解析文档，切分 `document_chunk`，生成向量写入 Milvus/Qdrant。  
4. 更新 `document.parse_status='PARSED'`，失败则 `FAILED + parse_error`。  
5. 聊天提问时写入 `message(role='user')`，Redis 缓存最近上下文。  
6. 检索向量库，按租户/知识库过滤，返回 chunk；生成回答写入 `message(role='assistant')`，并保存 `referenced_chunk_ids`。  
7. 音频提问：音频对象进 MinIO，转写文本入 `message.content`，必要时保存原音频文档记录。  
8. 图像问答：图像 embedding 写向量库，问题与图像联合检索后回答。

## 9. 安全与治理建议

- 所有查询必须携带 `tenant_id` 过滤，服务层强校验。
- MinIO 使用 STS 临时凭证或预签名 URL，避免长期密钥暴露。
- 敏感字段（如转写文本）可按租户策略加密或脱敏。
- 建议增加 `audit_logs` 记录关键操作（上传、删除、共享、导出）。

以上方案可支持工业级多租户、多模态 RAG 场景，并兼顾可扩展性、性能与可运维性。

## 10. Spring Cloud 微服务下的数据层最佳实践补充

1. 数据归属边界：每个微服务只直接写入自身主库表，跨服务通过 API 或事件总线交互，避免共享写入导致耦合。
2. 最终一致性：`document -> document_chunk -> vector` 建议通过 RocketMQ 发送分布式事务消息，保证底层落库一致性。
3. 幂等保障：解析与向量写入任务以 `document_id + chunk_index + embedding_model` 作为幂等键。
4. 读写分离与分区：高频消息表 `message` 可按时间分区，并结合只读副本支撑查询。
5. 观测与审计：关键数据变更记录 `trace_id`、`operator_id`、`tenant_id`，保证跨服务链路可追踪。
