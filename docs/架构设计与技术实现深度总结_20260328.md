## 3. 核心技术问题与解决方案

### 3.1 企业级消息存储策略

#### ❓ 问题

> "在企业常规做法中，都是将消息和会话存入数据库的吗？"

#### ✅ 结论

**是的，B2B 企业级应用通常采用全持久化策略。**

#### 📊 三种常见策略对比

| 策略 | 适用场景 | 优点 | 缺点 |
|------|----------|------|------|
| **全持久化** | B2B SaaS、金融、医疗 | 合规审计、数据分析、跨端同步 | 存储成本高 |
| **热冷分离** | C端大流量 | 平衡性能与成本 | 架构复杂 |
| **仅缓存** | 临时客服、匿名咨询 | 成本低、性能高 | 数据丢失风险 |

#### 🏢 企业案例

- **钉钉/飞书/Slack**：全量持久化，支持历史回溯
- **ChatGPT**：全量持久化，支持跨设备同步
- **客服机器人**：热数据 Redis + 冷数据 HBase

#### 💡 推荐方案

对于 **Guide-RAG** 项目：

```sql
-- ✅ 推荐：PostgreSQL 全持久化
CREATE TABLE message (
    id BIGSERIAL PRIMARY KEY,
    session_id BIGINT NOT NULL,
    role VARCHAR(20) NOT NULL,         -- user/assistant/system/tool
    content TEXT NOT NULL,
    content_type VARCHAR(20),
    referenced_chunk_ids JSONB,        -- 引用的文档块ID
    retrieval_context JSONB,           -- 检索上下文（含引用详情）
    gmt_create TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    gmt_modified TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    is_deleted SMALLINT DEFAULT 0
);

-- 创建 JSONB 字段的 GIN 索引
CREATE INDEX idx_message_retrieval_context ON message USING gin(retrieval_context);
```

**理由**：

1. 满足合规审计要求
2. 支持数据分析与模型优化
3. 实现跨端同步与历史回溯
4. PostgreSQL JSONB 性能优秀

### 3.2 多租户角色权限设计

#### ❓ 问题

> "Message 中的 role，user 表中并没有分具体的 role，请你评估在企业级开发中是否需要？"

#### ✅ 核心认知

**`Message.role` 和 `tenant_user.role` 是完全不同的概念！**

#### 🔍 两种 Role 的语义区别

| 字段 | 所在表 | 含义 | 枚举值 | 用途 |
|------|--------|------|--------|------|
| **message.role** | `message` | 对话角色 | user / assistant / system / tool | 标识消息的说话者身份 |
| **tenant_user.role** | `tenant_user` | 权限角色 | owner / admin / member | 控制用户在租户内的操作权限 |

#### 📐 架构设计

```
┌─────────────────────────────────────────────────────────┐
│                    多租户架构                            │
├─────────────────────────────────────────────────────────┤
│                                                          │
│  user 表（全局身份）                                      │
│  ├─ id, email, password                                 │
│  └─ 不存储 role（同一用户在不同租户有不同权限）            │
│                                                          │
│  tenant_user 表（租户内权限）                            │
│  ├─ tenant_id, user_id, role                            │
│  └─ role: owner/admin/member ← 权限角色                 │
│                                                          │
│  message 表（对话消息）                                  │
│  ├─ session_id, role, content                           │
│  └─ role: user/assistant/system/tool ← 对话角色          │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

#### 🌟 实际场景示例

**顾问咨询场景**：

```
张三（user_id = 1001）：
├─ 在"甲公司租户"(tenant_id = 2001) → role = "admin"   (权限角色)
├─ 在"乙公司租户"(tenant_id = 2002) → role = "member"  (权限角色)
└─ 在"自己公司租户"(tenant_id = 2003) → role = "owner"  (权限角色)

张三在甲公司发起对话：
├─ 消息1: role = "user"      (对话角色 - 用户提问)
├─ 消息2: role = "assistant" (对话角色 - AI回答)
└─ 消息3: role = "user"      (对话角色 - 用户追问)
```

#### ❌ 为什么 `user` 表不需要 role？

```sql
-- ❌ 错误设计（单租户思维）
CREATE TABLE user (
    id BIGINT PRIMARY KEY,
    email VARCHAR(100),
    role VARCHAR(20)  -- ← 无法表达"跨租户多角色"
);

-- ✅ 正确设计（多租户思维）
CREATE TABLE user (
    id BIGINT PRIMARY KEY,
    email VARCHAR(100)
    -- 不存储 role
);

CREATE TABLE tenant_user (
    tenant_id BIGINT,
    user_id BIGINT,
    role VARCHAR(20) DEFAULT 'member',  -- ← 租户内角色
    PRIMARY KEY (tenant_id, user_id)
);
```

#### 💪 CHECK 约束强化

#### ❓ 问题

> "添加 CHECK 约束强化有什么效果？"

#### ✅ 核心价值

CHECK 约束是**数据库层的最后一道防线**，提供以下保障：

| 防护场景 | 示例 | 效果 |
|---------|------|------|
| **防止 SQL 注入** | `INSERT INTO tenant_user VALUES (..., 'hacker')` | 被拒绝 |
| **防止代码 Bug** | Service 层传入 `role = "ADMIN"` (大写错误) | 被拒绝 |
| **防止数据迁移错误** | 脚本写错 `UPDATE ... SET role = 'superadmin'` | 被拒绝 |
| **防止工具误操作** | DBeaver 手动编辑数据时输入错误 | 被拒绝 |

#### 🛡️ 推荐实施

```sql
-- 为 tenant_user.role 添加约束
ALTER TABLE tenant_user 
ADD CONSTRAINT check_tenant_user_role 
CHECK (role IN ('owner', 'admin', 'member'));

-- 为 message.role 添加约束
ALTER TABLE message 
ADD CONSTRAINT check_message_role 
CHECK (role IN ('user', 'assistant', 'system', 'tool'));
```

#### 🔄 演进策略

当需要扩展角色时：

```sql
-- Step 1: 先删除旧约束
ALTER TABLE tenant_user DROP CONSTRAINT check_tenant_user_role;

-- Step 2: 添加新约束（包含新角色）
ALTER TABLE tenant_user 
ADD CONSTRAINT check_tenant_user_role 
CHECK (role IN ('owner', 'admin', 'member', 'viewer'));  -- 新增 viewer
```

### 3.4 引用信息（Citation）设计

#### ❓ 问题

> "CitationDTO 这个类是什么，我好像还没实现？"  
> "只需要 CitationDTO，不需要 Citation 吗？"

#### ✅ 设计决策

**当前阶段只需要 `CitationDTO`，不需要独立的 `Citation` 实体类。**

#### 📐 架构设计

```
┌─────────────────────────────────────────────────────────┐
│                   数据流向                               │
├─────────────────────────────────────────────────────────┤
│                                                          │
│  数据库层（Entity）                                       │
│  ├─ Message.retrievalContext (String)                   │
│  └─ 存储格式：JSONB 字符串                                │
│       {                                                  │
│         "citations": [                                   │
│           {                                              │
│             "docId": "doc_123",                          │
│             "chunkId": "chunk_456",                      │
│             "score": 0.92,                               │
│             "sourceText": "..."                          │
│           }                                              │
│         ],                                               │
│         "model": "bge-m3"                                │
│       }                                                  │
│                                                          │
│  服务层（Service）                                        │
│  ├─ toResDTO() 方法                                      │
│  └─ 解析 JSON → List<CitationDTO>                        │
│                                                          │
│  接口层（Controller）                                     │
│  ├─ MessageResDTO.citations                             │
│  └─ 返回给前端：结构化对象数组                             │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

#### 📝 代码实现

```java
// ✅ MessageResDTO.java（接口返回对象）
@Data
@ApiModel("消息响应对象")
public class MessageResDTO {
    
    @ApiModelProperty("消息ID")
    private Long messageId;
    
    @ApiModelProperty("角色：user/assistant/system/tool")
    private String role;
    
    @ApiModelProperty("消息内容")
    private String content;
    
    @ApiModelProperty("引用信息（仅assistant消息有）")
    private List<CitationDTO> citations;
    
    @ApiModelProperty("创建时间")
    private LocalDateTime gmtCreate;
    
    /**
     * 引用信息 DTO（静态内部类）
     * 不需要独立的 Entity，数据存储在 message.retrieval_context JSONB 字段
     */
    @Data
    @ApiModel("引用信息")
    public static class CitationDTO {
        
        @ApiModelProperty("文档ID")
        private String docId;
        
        @ApiModelProperty("文档块ID")
        private String chunkId;
        
        @ApiModelProperty("相似度分数")
        private Double score;
        
        @ApiModelProperty("引用原文")
        private String sourceText;
    }
}
```

#### 🔍 为什么用静态内部类？

| 设计选择 | 适用场景 | 优点 | 缺点 |
|---------|---------|------|------|
| **静态内部类** | Citation 仅在 MessageResDTO 中使用 | 代码内聚、命名空间清晰 | 不能独立复用 |
| **独立类** | Citation 在多个 DTO 中复用 | 易于复用 | 文件增多、可能过度设计 |

**当前选择**：静态内部类（因为 Citation 仅服务于 MessageResDTO）

#### 🚫 何时需要独立 Entity？

```sql
-- 如果需要以下能力，才考虑拆表：
-- 1. 独立查询引用信息
SELECT * FROM citation WHERE doc_id = 'doc_123';

-- 2. 引用统计分析
SELECT doc_id, COUNT(*) FROM citation GROUP BY doc_id;

-- 3. 外键强约束
ALTER TABLE citation ADD FOREIGN KEY (doc_id) REFERENCES document(id);
```

**当前阶段不需要，JSONB 存储足够。**

### 3.5 分页DTO转换机制

#### ❓ 问题

> "PageInfo<Session> pageInfo = new PageInfo<>(sessions);  
> PageInfo<SessionResDTO> result = new PageInfo<>();  
> 为什么要设置 result.setList(dtoList)？"

#### ✅ 核心理解

**PageInfo 的有参构造和无参构造行为完全不同！**

#### 🔍 PageHelper 工作原理

```java
// Step 1: PageHelper 启动分页（在 ThreadLocal 中设置参数）
PageHelper.startPage(2, 10);

// Step 2: 执行查询（返回的实际类型是 Page<Session>）
List<Session> sessions = sessionMapper.selectByTenantIdAndUserId(1001, 1001);

// Step 3: Page 类的特殊性
sessions.getClass().getName();  // "com.github.pagehelper.Page"
sessions instanceof ArrayList;   // true（Page 继承自 ArrayList）
sessions instanceof Page;        // true
((Page) sessions).getTotal();    // 35（总记录数）

// Step 4: Stream 转换后丢失了 Page 类型
List<SessionResDTO> dtoList = sessions.stream()
        .map(this::toResDTO)
        .collect(Collectors.toList());

dtoList.getClass().getName();   // "java.util.ArrayList"
dtoList instanceof Page;         // false（丢失了 Page 类型）
// ((Page) dtoList).getTotal();  // ❌ ClassCastException
```

#### 📊 内存状态对比

##### 场景1：有参构造（传入 sessions）

```java
PageInfo<Session> pageInfo = new PageInfo<>(sessions);

// 内部逻辑：
if (sessions instanceof Page) {  // ✅ 条件成立
    Page page = (Page) sessions;
    this.pageNum = page.getPageNum();    // 2
    this.total = page.getTotal();        // 35
    this.pages = page.getPages();        // 4
    this.list = sessions;  // [Session1, Session2, ...]
}

// 结果：
pageInfo.getList();     // [Session1, Session2, ...] ✅
pageInfo.getTotal();    // 35 ✅
pageInfo.getPageNum();  // 2 ✅
```

##### 场景2：无参构造（空对象）

```java
PageInfo<SessionResDTO> result = new PageInfo<>();

// 内部逻辑：
// ← 什么都不做

// 结果：
result.getList();     // null ❌
result.getTotal();    // 0 ❌
result.getPageNum();  // 0 ❌
```

##### 场景3：有参构造传入 dtoList（无效）

```java
// ❌ 错误做法
PageInfo<SessionResDTO> result = new PageInfo<>(dtoList);

// 内部逻辑：
if (dtoList instanceof Page) {  // ❌ 条件不成立（dtoList 是 ArrayList）
    // 这段代码不会执行
}
this.list = dtoList;  // 只设置了 list

// 结果：
result.getList();     // [SessionResDTO1, ...] ✅
result.getTotal();    // 0 ❌ 丢失了分页元数据
result.getPageNum();  // 0 ❌ 丢失了分页元数据
```

#### ✅ 正确做法

```java
// Step 1: 从 Page<Session> 中提取分页元数据
PageInfo<Session> pageInfo = new PageInfo<>(sessions);

// Step 2: 转换 Entity → DTO
List<SessionResDTO> dtoList = sessions.stream()
        .map(this::toResDTO)
        .collect(Collectors.toList());

// Step 3: 创建新的 PageInfo<DTO>
PageInfo<SessionResDTO> result = new PageInfo<>();

// Step 4: 手动设置 list 和分页元数据
result.setList(dtoList);                        // ← 设置 DTO 列表
result.setTotal(pageInfo.getTotal());           // ← 拷贝总记录数
result.setPageNum(pageInfo.getPageNum());       // ← 拷贝当前页码
result.setPageSize(pageInfo.getPageSize());     // ← 拷贝每页大小
result.setPages(pageInfo.getPages());           // ← 拷贝总页数
result.setStartRow(pageInfo.getStartRow());     // ← 拷贝起始行
result.setEndRow(pageInfo.getEndRow());         // ← 拷贝结束行
result.setPrePage(pageInfo.getPrePage());       // ← 拷贝上一页
result.setNextPage(pageInfo.getNextPage());     // ← 拷贝下一页
result.setIsFirstPage(pageInfo.isIsFirstPage()); // ← 拷贝是否首页
result.setIsLastPage(pageInfo.isIsLastPage());   // ← 拷贝是否末页
result.setHasPreviousPage(pageInfo.isHasPreviousPage()); // ← 拷贝是否有上一页
result.setHasNextPage(pageInfo.isHasNextPage()); // ← 拷贝是否有下一页
```

#### 💡 优化建议：抽取工具类

```java
// ✅ PageUtils.java（放在 rag-common 模块）
package com.guiderag.common.utils;

import com.github.pagehelper.PageInfo;
import java.util.List;
import java.util.function.Function;
import java.util.stream.Collectors;

/**
 * 分页工具类
 * 
 * @author guide-rag
 */
public class PageUtils {
    
    /**
     * PageInfo<Entity> 转换为 PageInfo<DTO>
     * 
     * @param pageInfo 原始分页对象
     * @param mapper   转换函数 (Entity -> DTO)
     * @return 转换后的分页对象
     */
    public static <E, D> PageInfo<D> convert(PageInfo<E> pageInfo, Function<E, D> mapper) {
        // 1. 转换 list
        List<D> dtoList = pageInfo.getList().stream()
                .map(mapper)
                .collect(Collectors.toList());
        
        // 2. 创建新的 PageInfo
        PageInfo<D> result = new PageInfo<>();
        
        // 3. 设置 list
        result.setList(dtoList);
        
        // 4. 拷贝分页元数据
        result.setTotal(pageInfo.getTotal());
        result.setPageNum(pageInfo.getPageNum());
        result.setPageSize(pageInfo.getPageSize());
        result.setPages(pageInfo.getPages());
        result.setStartRow(pageInfo.getStartRow());
        result.setEndRow(pageInfo.getEndRow());
        result.setPrePage(pageInfo.getPrePage());
        result.setNextPage(pageInfo.getNextPage());
        result.setIsFirstPage(pageInfo.isIsFirstPage());
        result.setIsLastPage(pageInfo.isIsLastPage());
        result.setHasPreviousPage(pageInfo.isHasPreviousPage());
        result.setHasNextPage(pageInfo.isHasNextPage());
        result.setNavigatepageNums(pageInfo.getNavigatepageNums());
        result.setNavigatePages(pageInfo.getNavigatePages());
        result.setNavigateFirstPage(pageInfo.getNavigateFirstPage());
        result.setNavigateLastPage(pageInfo.getNavigateLastPage());
        
        return result;
    }
}
```

### 4.1 toResDTO 方法深度解析

#### ❓ 问题

> "解析这段代码的作用？"

```java
private MessageResDTO toResDTO(Message message) {
    MessageResDTO dto = new MessageResDTO();
    dto.setMessageId(message.getMessageId());
    dto.setRole(message.getRole());
    dto.setContent(message.getContent());
    dto.setContentType(message.getContentType());
    dto.setGmtCreate(message.getGmtCreate());

    // 解析引用信息（仅 assistant 消息）
    if ("assistant".equals(message.getRole()) && StringUtils.hasText(message.getReferencedChunkIds())) {
        try {
            // retrievalContext 中可能存储更详细的引用信息
            String contextJson = message.getRetrievalContext();
            if (StringUtils.hasText(contextJson) && !"{}".equals(contextJson)) {
                JSONObject context = JSON.parseObject(contextJson);
                JSONArray citationsArray = context.getJSONArray("citations");
                if (citationsArray != null && !citationsArray.isEmpty()) {
                    List<MessageResDTO.CitationDTO> citations = new ArrayList<>();
                    for (int i = 0; i < citationsArray.size(); i++) {
                        JSONObject citObj = citationsArray.getJSONObject(i);
                        MessageResDTO.CitationDTO citation = new MessageResDTO.CitationDTO();
                        citation.setDocId(citObj.getString("docId"));
                        citation.setChunkId(citObj.getString("chunkId"));
                        citation.setScore(citObj.getDouble("score"));
                        citation.setSourceText(citObj.getString("sourceText"));
                        citations.add(citation);
                    }
                    dto.setCitations(citations);
                }
            }
        } catch (Exception e) {
            // 解析失败时忽略引用信息
            dto.setCitations(null);
        }
    }

    return dto;
}
```

```java
            JSONObject context = JSON.parseObject(contextJson);
            
            // 4. 提取 citations 数组
            JSONArray citationsArray = context.getJSONArray("citations");

        // ========================================
```

### 5.4 不要携带 BOM 前缀

#### ❓ 问题

> "为什么你在命令行输出的文字我复制到 IDEA 之后开头会多一串字母 ZWNBSP？"

#### ✅ 原因与解决

**BOM (Byte Order Mark)** 是 UTF-8 文件的可选标识符，但在 Java 中会导致问题：

```java
// ❌ 带 BOM 的文件（开头有隐藏字符 \uFEFF）
public class MyClass {  // ← 编译错误：非法字符

// ✅ 不带 BOM 的文件
public class MyClass {  // ← 正常
```

**解决方案**：

1. IDEA 设置：`Settings → Editor → File Encodings`
   - `Global Encoding: UTF-8`
   - `Project Encoding: UTF-8`
   - ✅ 取消勾选 `Create UTF-8 files with BOM`

2. 批量转换：
```bash
# 使用 dos2unix 工具
find . -name "*.java" | xargs dos2unix
```

```java
package com.guiderag.common.utils;

import com.github.pagehelper.PageInfo;

import java.util.List;
import java.util.function.Function;
import java.util.stream.Collectors;

/**
 * 分页工具类
 * <p>
 * 解决 PageInfo<Entity> 转 PageInfo<DTO> 的样板代码问题
 *
 * @author guide-rag
 */
public class PageUtils {

    /**
     * PageInfo 类型转换
     * <p>
     * 使用示例：
     * <pre>
     * PageInfo<Session> pageInfo = new PageInfo<>(sessions);
     * return PageUtils.convert(pageInfo, this::toResDTO);
     * </pre>
     *
     * @param pageInfo 原始分页对象
     * @param mapper   转换函数 (Entity -> DTO)
     * @param <E>      Entity 类型
     * @param <D>      DTO 类型
     * @return 转换后的分页对象
     */
    public static <E, D> PageInfo<D> convert(PageInfo<E> pageInfo, Function<E, D> mapper) {
        // 1. 转换 list
        List<D> dtoList = pageInfo.getList().stream()
                .map(mapper)
                .collect(Collectors.toList());

        // 2. 创建新的 PageInfo
        PageInfo<D> result = new PageInfo<>();

        // 3. 设置 list
        result.setList(dtoList);

        // 4. 拷贝分页元数据（完整版）
        result.setTotal(pageInfo.getTotal());
        result.setPageNum(pageInfo.getPageNum());
        result.setPageSize(pageInfo.getPageSize());
        result.setPages(pageInfo.getPages());
        result.setStartRow(pageInfo.getStartRow());
        result.setEndRow(pageInfo.getEndRow());
        result.setPrePage(pageInfo.getPrePage());
        result.setNextPage(pageInfo.getNextPage());
        result.setIsFirstPage(pageInfo.isIsFirstPage());
        result.setIsLastPage(pageInfo.isIsLastPage());
        result.setHasPreviousPage(pageInfo.isHasPreviousPage());
        result.setHasNextPage(pageInfo.isHasNextPage());
        result.setNavigatepageNums(pageInfo.getNavigatepageNums());
        result.setNavigatePages(pageInfo.getNavigatePages());
        result.setNavigateFirstPage(pageInfo.getNavigateFirstPage());
        result.setNavigateLastPage(pageInfo.getNavigateLastPage());

        return result;
    }
}
```
